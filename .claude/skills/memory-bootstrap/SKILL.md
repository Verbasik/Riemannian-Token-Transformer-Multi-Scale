Исследуй проект и первично заполни memory bank. Выполни шаги точно.

## 1. Оцени состояние memory bank

Прочитай `memory-bank/PROJECT.md`.
- Если содержит `[Название проекта]` — память не инициализирована, продолжай все шаги.
- Если частично заполнен — пропусти уже заполненные поля, заполни только пустые.
- Если полностью заполнен — остановись и сообщи: «Memory bank уже инициализирован. Используй /memory-audit для проверки.»

## 2. Исследуй проект

Читай в таком порядке:

**Корневой контекст:**
- `README.md` (или `readme.md`)
- `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` / `pom.xml` — первый найденный
- `.env.example`, `docker-compose.yml`, `Makefile` — если существуют

**Структура:**
- Директории глубиной 2 от корня
- Первые 20 файлов в `src/`, `app/`, `lib/`, `cmd/` — если существуют

**Тесты:**
- `test/`, `tests/`, `__tests__/`, `spec/` — если существуют
- Конфигурационные файлы тестов: `jest.config.*`, `pytest.ini`, `vitest.config.*`

**Стиль кода:**
- `.prettierrc*`, `.eslintrc*`, `pyproject.toml [tool.ruff]`, `.editorconfig`

Цель: установить стек, тип проекта, ключевые модули, тестовый фреймворк, coding style.

## 3. Заполни PROJECT.md

Запиши в `memory-bank/PROJECT.md`:
- **Название**: из package.json / README / имени директории
- **Тип**: SaaS / CLI / API / библиотека / монорепо — определи по структуре
- **Стадия**: MVP если нет тестов/CI; beta если есть тесты; production если есть CI + changelog
- **Репозиторий**: из package.json `repository` или `.git/config` remote
- **Миссия**: 2–3 предложения на основе README
- **Границы**: в scope / вне scope — на основе README и структуры
- **Словарь домена**: ключевые термины из README и кода (не менее 3, если определяемо)
- **Стейкхолдеры**: если указаны в README/CONTRIBUTING
- Добавь: `Верифицировано: [сегодня]`

Не заполняй поля, если не можешь установить достоверно — оставь `[требует верификации]`.

## 4. Заполни ARCHITECTURE.md

Запиши в `memory-bank/ARCHITECTURE.md`:
- **System overview**: как система работает на верхнем уровне (2–4 предложения)
- **Component map**: ASCII-диаграмма на основе реальной структуры директорий
- **Modules**: таблица с реальными модулями (не шаблонными `[module]`)
- **External dependencies**: из package manifest с версиями
- **Infrastructure**: только если определяемо из конфига (docker-compose, Makefile и т.п.)
- Добавь: `last_verified: [сегодня]`

## 5. Заполни CONVENTIONS.md (если определяемо)

Только если можно установить достоверно из конфигов или кода:
- Язык, рантайм, форматтер
- Стиль именования (camelCase / snake_case / PascalCase)
- Структура файлов

Если конфигов нет — оставь плейсхолдеры и добавь вопрос в OPEN_QUESTIONS.md.

## 6. Заполни TESTING.md (если определяемо)

Только если тестовый фреймворк обнаружен:
- Команды запуска тестов
- Расположение тестов
- Типы тестов (unit / integration / e2e)

Если тестов нет — добавь вопрос в OPEN_QUESTIONS.md.

## 7. CONSTITUTION.md — только предложение

**НЕ записывай в CONSTITUTION.md напрямую.**

На основе типа и стека проекта сформулируй 3–5 предлагаемых принципов и выведи их в финальном отчёте под заголовком:

```
⚠️ ПРЕДЛАГАЕМЫЕ ПРИНЦИПЫ ДЛЯ CONSTITUTION.md (требуют ревью и одобрения человека):
```

## 8. Создай .local файлы

Создай `memory-bank/.local/CURRENT.md`:

```markdown
# Текущий контекст

Последнее обновление: [ГГГГ-ММ-ДД] — memory-bootstrap

## Статус
Memory bank первично инициализирован из кодовой базы проекта.
Базовый контекст заполнен. Требует верификации человеком.

## Следующие шаги
1. Проверить и дополнить CONSTITUTION.md (предложенные принципы — в отчёте bootstrap)
2. Уточнить пробелы из OPEN_QUESTIONS.md
3. Запустить /memory-audit для проверки целостности

## Активные файлы
- memory-bank/PROJECT.md
- memory-bank/ARCHITECTURE.md

## Checkpoint
[ГГГГ-ММ-ДД]
```

Создай `memory-bank/.local/HANDOFF.md`:
- **Что сделано**: memory bank первично инициализирован через memory-bootstrap; заполнены PROJECT.md, ARCHITECTURE.md и частично другие файлы
- **Что дальше**: заполнить CONSTITUTION.md вручную, закрыть open questions
- **Осторожно**: все записи верифицированы только поверхностно — требуют подтверждения человеком перед использованием как canonical source

## 9. Зафиксируй неопределённости

Всё, что не удалось установить достоверно, добавь в `memory-bank/OPEN_QUESTIONS.md`:

| # | Вопрос | Контекст | Поднят | Владелец | Статус |
|---|--------|----------|--------|----------|--------|
| N | [конкретный вопрос] | memory-bootstrap: не определено автоматически | [сегодня] | — | Открыт |

## 10. Отчёт

```
## Результат memory-bootstrap — [дата]

### Заполнено
- PROJECT.md: [что заполнено]
- ARCHITECTURE.md: [что заполнено]
- CONVENTIONS.md: [заполнено / пропущено — причина]
- TESTING.md: [заполнено / пропущено — причина]
- .local/CURRENT.md: создан
- .local/HANDOFF.md: создан

### Пропущено (не определено автоматически)
- [список полей]

### Открытые вопросы добавлены: [N]

### Требуют ревью человека
- CONSTITUTION.md — предлагаемые принципы:
  I. [принцип]
  II. [принцип]
  ...

### Следующий шаг
Запусти /memory-audit для проверки целостности.
```

## Правила

- Никогда не записывай секреты, токены или PII.
- Не перезаписывай поля с существующим (не шаблонным) контентом.
- Неподтверждённые факты помечай `[требует верификации]`, не выдавай за canonical.
- CONSTITUTION.md — только предлагай в отчёте, не пиши напрямую.
- Каждый заполненный факт получает `Верифицировано: [дата]`.
- Жёсткий лимит CURRENT.md: 80 строк. HANDOFF.md: 40 строк.
